# 科研项目管理平台客户展示 Deck 叙事 Spec

> 日期：2026-07-06

> 当前版本：v09，21 页客户展示 Deck。

## 1. 交付物定位

该 HTML 是面向科研项目管理平台建设沟通的客户展示 Deck，不是内部开发说明，也不是网页长页。它要完成四件事：

- 说明科研项目管理的核心要点：对象完整、流程清楚、权责明确、风险可见、材料齐套、结果可用。
- 说明需求框架已经形成清晰合理的建设基础，可以转译为平台模块、流程规则、台账字段和校验条件。
- 展示平台落地形态：一本账、差异化流程、四色预警、变更门控、经费合同、验收闸门、成果转化、后评价和看板分析。
- 说明升维路径：流程记录沉淀为过程证据，科研项目对象关系固化为本体模型，AI 在依据、规则、权限和人工复核内工作。

## 2. 受众与视角

受众是科研项目管理平台建设的业务管理人员、执行单位、信息化、财务合同、档案和专业系统相关部门。页面采用客观视角：

- 不使用“贵方”“我们”“客户”“甲方”“乙方”等关系称谓。
- 不出现“本版”“本轮”“这份需求”等内部改稿话术。
- 不用“普通平台”“只是填表”“只是做页面”等贬低对比。
- 不把外部公司作为正文主线，外部资料只作为方法参照。
- AI 从建设第一天进入平台，但所有输出绑定依据、规则、权限、人工复核和审计留痕。

## 3. 主线

Deck 采用“管理要点 → 需求分析 → 样例项目 → 平台落地 → 过程证据 → 本体与 AI → 集成路线”的主线。

核心主张：

> 科研项目管理平台的核心，不只是记录项目状态，而是把项目、计划、经费、合同、材料、成果、评价和审批组织成一条可追溯、可审核、可复用的管理链。

## 5. 需求覆盖锚点

后续修改必须继续对照 `docs/specs/customer-requirement-coverage-matrix.md` 和 `docs/specs/customer-deck-v02-brief.md`。增强能力必须回到需求文档中的管理对象、流程规则、角色权限、材料要求或系统边界。

必须覆盖：

- 建设定位：统筹归集、宏观管控、不干预具体执行计划、科研项目一本账。
- 项目分类：联邦级、洲级、公司级；渠道编码全局唯一。
- 角色权限：项目团队、责任总师、管理团队、财务团队、超级管理员。
- 预警规则：绿、蓝、黄、红四色状态；节点到期前 30 天预警；站内消息、企业邮箱、蓝信推送。
- 生命周期：项目总览、立项、实施、验收、后评价、成果转化。
- 实施规则：里程碑、计划管理系统同步、经费预算、里程碑绑定核销、项目变更、数据变更。
- 验收规则：里程碑闭环、外协交付物验收、节点经费匹配核销完成后再提交验收。
- 结果闭环：交付物、成果包、成果编号、协作单位评价、成果转化台账、后评价。
- 系统集成：计划管理系统、财务平台、合同采购系统、审批系统、档案系统、DOORS、CMOS 等按边界读写或引用。

## 6. 视觉与文案原则

- 每页一个视觉主角，图示必须承载业务关系，不放无意义元素。
- 内容页统一柔和纸色，封面和收束页使用深色；不通过随机背景色制造章节感。
- 产品 mockup 和 SVG 图示优先表达对象、流程、规则、证据、动作和数据流。
- 生成图只表达业务关系、复杂结构和视觉节奏；关键文字、字段、规则和流程必须由 HTML/SVG 精确呈现。
- 文案使用短结论、并列短句和对象名；不写文章摘句、设问句和讲稿句。
- 每页必须有主张、机制、证据或样例三层信息；不能只放标题和少量短卡片。
- 任何 AI 能力都必须回答：依据来自哪里、谁复核、动作进入哪里、如何留痕。
- 任何系统集成都必须回答：读什么、写什么、谁确认、失败如何处理。
